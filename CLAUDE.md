# CLAUDE.md

このリポジトリは、Flutter で作る個人用モバイル Sokoban アプリです。

## 最初に読むもの

作業を始めるときは、次の順で読むこと:
1. `docs/product-brief.md`
2. `docs/product-backlog.md`
3. `docs/architecture.md`
4. この `CLAUDE.md`

これらの文書とタスク内容が衝突する場合は、まず backlog と architecture を優先し、広い変更をする前に確認すること。

## プロジェクトの意図

- iOS / Android 向けの、小さくても磨かれた Sokoban アプリを作る。
- 早すぎる複雑化より、着実で理解しやすい進行を優先する。
- 人間にも AI にも安全に変更しやすいコードベースを保つ。

## タスク選択ルール

AI エージェントは次をしてよい:
- スコープの明確な機能実装
- バグ修正
- テストの追加・更新
- 作業の issue / PR 単位への分解
- PR 説明文の下書き

AI エージェントは、基本的に `ready` と明示された backlog 項目、またはそれと同等に明確なタスクだけを自走で選ぶこと。

AI エージェントは次をしてはいけない:
- 明示的な承認なしに広いリファクタを始める
- 明確な理由なしに大きな依存を追加する
- 推測でプロダクト方針を変える
- 実装の都合に合わせて architecture 文書を書き換えて辻褄を合わせる

## 実装ルール

- 変更は小さく、レビューしやすく保つ。
- 過度に凝った抽象化より、シンプルな Flutter パターンを優先する。
- 合成しやすさと読みやすい状態遷移を重視する。
- 可能な限り、コアのパズルロジックを UI から分離する。
- ドメインロジックは widget に強く結び付けず、テストしやすく保つ。
- プラットフォーム差分が意図的でない限り、iOS / Android 間の挙動はそろえる。

## 依存ポリシー

- 新しい依存は必要最小限にする。
- Flutter / Dart 標準機能、または広く使われている安定パッケージを優先する。
- 依存追加がアーキテクチャ、アプリサイズ、状態管理、永続化、ナビゲーションに大きく影響する場合は、先に確認すること。

## テスト方針

- 可能な限り、変更に応じてテストを追加・更新する。
- とくに、盤面ロジック、移動ルール、押し出し判定、レベル解析、クリア判定、再発バグの防止を優先してテストする。
- UI の重要な挙動については、安定していて価値が高い範囲で widget test を追加してよい。
- テストを省略する場合は、その理由を明記すること。

## PR の期待値

1 つの PR では、基本的に 1 つのまとまりある改善を扱うこと。

PR 下書きには次を含める:
- 何を変えたか
- なぜ変えたか
- 追加・更新したテスト
- 手動での確認方法
- 残っている懸念や次の課題

## 禁止事項

次はしないこと:
- 承認なしの大規模リファクタ
- 不要な依存追加
- 文書を更新せずに backlog 項目の意味を変えること
- テストしていないのに通ったと主張すること
- 書かれていない要件を勝手に確定事項として扱うこと

## 作業モデル

この repo では軽量な計画モデルを使う:
- backlog bucket: `Now`, `Later`, `Icebox`
- item status: `ready`, `needs-clarification`, `blocked`, `done`

迷ったら、勝手に決め打ちするよりも、文書を軽く更新して 1 つだけ具体的な確認を出すほうを優先すること。

## ドキュメントと言語

- このリポジトリ内のドキュメント、計画、PR 下書き、設計メモは基本的に **日本語** で作成すること。
- コード中の識別子、API 名、ライブラリ名、Flutter / Dart の構文は通常どおり英語でよい。
- 外部仕様やライブラリ由来の用語は必要に応じて英語を併記してよいが、本文の説明は日本語を優先する。
