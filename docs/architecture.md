# アーキテクチャ

> 個人用 Flutter Sokoban アプリの初期アーキテクチャメモ。
>
> 必要以上に重くせず、最初は短く保つ。

## 目標

- iOS / Android 向けの、遊べる Sokoban アプリを出す。
- パズルロジックを正しく、テストしやすく保つ。
- AI 支援で反復開発しやすい、素直な UI / コード構造にする。
- v1 の段階で過剰設計せず、必要になったときに伸ばせる形にする。
- v1 では複数ステージ、画面内ボタン操作、Undo / Restart を成立させる。

## 設計原則

### 1. ロジックを先に、UI を後に
Sokoban のコアルールは、できるだけ widget の中ではなく plain Dart code に置く。

### 2. 単純な構造を優先する
責務が分かれる最小限の構造から始める。重いパターンは早すぎる段階で持ち込まない。

### 3. 重要な振る舞いをテストする
移動ルール、箱押しルール、盤面妥当性、クリア判定は unit test しやすく保つ。

### 4. デフォルトはクロスプラットフォーム
意図した差分でない限り、iOS / Android で挙動をそろえる。

### 5. 変更は小さくレビュー可能に
広い書き換えより、段階的な前進を優先する。

## ドメイン境界

### Domain
担当するもの:
- タイル定義
- 盤面表現
- プレイヤー移動
- 箱押し判定
- クリア判定

求める性質:
- なるべく framework-light
- 決定的に動く
- 実用的な範囲で unit test を厚めにしやすい

### Presentation
担当するもの:
- 盤面描画
- 入力処理
- 状態表示
- クリア演出 / メッセージ
- モバイルでの見やすいレイアウト

コアのパズルルールを直接抱え込まないこと。

## 状態管理

最初から state management library を必須にしない。

開始時点の方針:
- まずは Flutter 標準機能を優先
- 複雑さが実際に増えたら、その時点で導入を検討する

状態管理が窮屈になってきたときは、惰性で採用せず、何が困っているかを確認してから選ぶ。

## 依存制約

- 依存は最小限に保つ。
- 安定していて広く使われているパッケージを優先する。
- アーキテクチャ全体を変えうる依存には慎重になる。

通常、事前相談したほうがよい例:
- 状態管理フレームワーク
- 実際の必要以上に大きいルーティングフレームワーク
- 永続化 / ストレージ系パッケージ
- アナリティクス / クラッシュ収集系

## テスト方針

基本ライン:
- コアパズルロジックには unit test
- 必要価値の高いユーザー操作には widget test
- 些細な UI 詳細を固定しすぎる brittle なテストは避ける

多くの変更では、可能であればテストを追加または更新する。

## 今はやらないこと

v1 を絞るため、初期段階では次に強くコミットしない:
- マルチプレイ / ソーシャル機能
- クラウド同期
- 高度なアニメーション基盤
- plugin-heavy な構成
- アプリ内エディタ機能

## 未解決の論点

現時点では、次はあえて未確定にしておく:
- レベルデータの正確な表現形式
- ジェスチャー、画面内ボタン、キーボード操作のどれをどこまで支えるか
- 進捗や設定の永続化戦略
- 専用の application layer が早期に必要かどうか
