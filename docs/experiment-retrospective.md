# 実験振り返りメモ

> OpenClaw / Claude Code を使った本アプリ開発実験の、短い振り返り。
>
> 主に「なぜ UI 調整が長引いたか」「なぜ複数ステージ実装まで進みにくかったか」を整理する。

## この文書の目的

この文書は、今回の開発実験で見えた進め方の癖と課題を、次に活かせる形で残すためのメモである。

とくに次を対象にする。

- UI 改善が具体的な指示待ちになりやすかった理由
- 細かい UI 調整が続き、機能追加へ進みにくかった理由
- 次回、同じ停滞を避けるための運用上の学び

## 観測された課題

今回、目立った課題は大きく 2 つあった。

### 1. UI 改善が具体指示待ちになりやすかった

機能面では前進していても、見た目については「仮アイコン感」「未完成感」が長く残った。

つまり、
- 動くことは動く
- しかし見た目の完成度は十分に上がらない
- 人間が違和感を言語化してから改善が進む

という流れになりやすかった。

### 2. UI の細かい改善が続き、複数ステージ対応まで進みにくかった

盤面、カード、アイコン、モバイル余白などの細かい改善は積み上がった一方で、
複数ステージの実装や進行管理のような、少し大きめの機能には着手が遅れた。

## 要因の整理

### 1. UI の良し悪しは機能より曖昧である

ロジックやテストは、正誤を比較的判断しやすい。
一方で UI は、

- 何が仮っぽく見えるか
- どこまで整えば十分か
- どの方向の見た目が望ましいか

が曖昧になりやすい。

そのため、AI 側はまず安全寄りに、

- 壊さない
- 小さく直す
- 既存部品で済ませる

方向へ寄りやすかった。

結果として、「機能的には成立しているが、見た目はまだ仮っぽい」という中間状態で止まりやすかった。

### 2. 小さくレビュー可能な変更を優先する運用だった

今回の運用では、次を強く重視していた。

- 1 回に 1 つの改善
- 小さくレビュー可能な変更
- 広いリファクタを避ける
- 安全に進める

これは壊れにくい一方で、UI のように複数要素をまとめて整えたほうがよい領域では、
改善が分割されすぎる副作用があった。

結果として、
- 盤面だけ直す
- 次にボタンだけ直す
- 次にアイコンだけ直す
- さらに別のアイコンだけ直す

と、改善が細切れになりやすかった。

### 3. 「見た目改善を最優先」にした影響が大きかった

途中から backlog と自動改善の指示で、見た目改善を明示的に優先した。

これはユーザーの違和感に素直に反応した結果として自然だったが、同時に

- 複数ステージ
- 進行管理
- 次ステージ導線

のような機能実装よりも、
UI の細かい改善タスクが選ばれやすくなった。

つまり、運用ルール上も UI 改善へ強く重心が寄っていた。

### 4. UI 改善の終了条件が先に定義されていなかった

今回の UI 改善では、

- どこまで直したら UI 改善フェーズを抜けるか
- 仮アイコン感がどの状態で解消とみなせるか
- 何回くらい改善したら次の機能へ進むか

といった終了条件が明文化されていなかった。

このため UI 改善は「やればやるほど次の違和感が見つかる」領域になり、
結果として長引きやすかった。

### 5. 複数ステージ対応は、UI 調整より塊が大きいタスクだった

複数ステージ対応には少なくとも、

- 複数レベル読込
- 現在ステージ管理
- 遷移導線
- クリア後進行
- テスト更新

が必要になる。

これは 1 つ 1 つの UI 微修正より、明らかにタスクの粒が大きい。

そのため、
- 小さく終わりやすい UI 修正
- やや大きい複数ステージ対応

が並んだとき、前者が選ばれやすかった。

### 6. 人間のフィードバックが UI の違和感を強く示した

これは悪いことではなく、むしろ重要な観測だった。

ただし、
- 仮アイコンっぽい
- スマホで盤面が小さい
- まだ未完成に見える

のような具体的フィードバックがあると、AI はそこに素直に最適化しやすい。

結果として、優先順位としては自然でも、全体進行としては UI 調整ループに入りやすかった。

## まとめると何が起きていたか

今回の停滞は、AI の能力不足だけで起きたというより、
次の組み合わせで起きたと考えるのが自然である。

- UI 品質の判断が曖昧だった
- 小さく安全に進める運用だった
- 見た目改善を最優先にした
- UI 改善の終了条件がなかった
- 複数ステージ対応のほうがタスクの塊として重かった

その結果、
**細かい UI 改善を積み続けるほうへ流れやすかった**。

## 次回に向けた運用上の学び

### 1. UI 改善には終了条件を置く

たとえば次のように、UI フェーズの抜け条件を先に決めておくとよい。

- 仮アイコンをすべて除去したら次へ進む
- スマホで盤面領域が一定以上になったら一旦区切る
- UI 改善 PR を 2 〜 3 本までに制限する

### 2. 機能マイルストーンを固定する

たとえば、

- 次は複数ステージ対応が終わるまで UI 微調整は禁止
- 複数ステージ playable を作ってから次の磨き込みへ戻る

のように、一定期間は機能優先と明示したほうが進みやすい。

### 3. UI と機能で粒度ルールを分ける

UI は 1 要素ずつではなく、ある程度まとまりで直したほうがよい場合がある。

そのため、
- 機能追加は小さくレビュー可能に刻む
- UI 改善は画面単位・導線単位でまとめて扱う

のように、領域ごとにルールを変える価値がある。

### 4. 「次に完成させる体験」を先に固定する

今回でいえば、

- 複数ステージを遊べる
- クリア後に次へ進める
- Pages でそのまま試せる

までを 1 つのまとまった目標として固定したほうが、判断がぶれにくかった。

### 5. docs の同期を早めに行う

README や backlog が現状とずれると、
人間も AI も「次に何をやるべきか」を見誤りやすい。

小さくても、docs 同期は判断精度を上げる効果がある。

## この実験で得られた前向きな点

課題だけでなく、次の点はうまく機能した。

- 小さな PR を積む運用自体は安全だった
- GitHub Pages 公開まで通せた
- Flutter Web 上で遊べる形までは比較的早く到達できた
- UI の違和感をフィードバックすると、その方向への改善は実際に進んだ
- backlog を使った優先順位調整は機能した

つまり、土台や運用そのものが失敗だったというより、
**優先順位と終了条件の設計が次の改善点**だったと考えるのがよい。

## 最後に

次回同様の実験をするときは、次の形が現実的である。

1. まず完成させたい機能マイルストーンを決める
2. UI 改善は回数または終了条件を先に決める
3. backlog に「今は何を優先するか」を強く書く
4. docs を定期的に同期する

この 4 点を押さえるだけでも、今回のような
「UI は整うが、次の大きい機能へ進みにくい」状態はかなり減らせるはずである。
